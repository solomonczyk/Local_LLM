## 1) Operating mode (как работаем)
- Формат ответов/изменений: сначала **Conclusions**, затем **Details**, затем **Risks**, затем **Improvements**, затем **Next Step**.
- Не переписывать код без запроса.
- Делать изменения минимальными (small diffs), обратимыми.
- Перед любым изменением: коротко описать план и затронутые файлы.

## 2) Архитектурные принципы
- Модульность > монолитные правки.
- Явные интерфейсы/контракты между сервисами.
- Конфиг через env / config файлы, не хардкод.
- Логи/метрики/трассировка — обязательны для продовых компонентов.

## 3) Quality gates (качество)
- Форматирование: editorconfig обязателен.
- Линт/тесты должны проходить перед merge.
- Любая новая фича: хотя бы 1 тест (unit/integration — по смыслу).
- Ошибки: стандартизировать (code/message/details).

## 4) Security & privacy
- Секреты не коммитить (ключи, токены, .env).
- Минимизировать отправку данных в облако.
- Логи не должны содержать персональные данные/секреты.

## 5) Repository conventions (соглашения)
- Структура:
  - /docs — документация и правила
  - /agent_runtime/kb — базы знаний агентов (если есть)
  - /docker — контейнеры и окружение (если есть)
- Именование:
  - файлы: snake_case
  - классы: PascalCase
  - функции: snake_case / camelCase (по языку проекта)

## 6) Decision records (решения)
- Важные решения фиксируем в /docs/adr/ (Architecture Decision Records).
- Формат ADR: Context → Decision → Consequences.
